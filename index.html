§ How to use this guide

Read top-to-bottom once. After that, jump to the interaction type you're building.

Each interaction type has the same four parts, in this order:

  1. Plain-language name & when to use it — how to describe it in a ticket or to a teacher.
  2. The intended look — the best real example, embedded, with a link to open it live.
  3. The spec that must be true every time — exact colors, sizes, layout, validation, lock-down.
  4. How to conform a non-standard screen — a step-by-step you can do by hand or run as a script. Script steps link straight to the tool that performs them.

"Should be there every time" items are written as a checklist. A screen that fails any checklist item is non-conforming and is a candidate for the auto-fix pass.

The one thing to internalize
The ELA templates and the K-5 math handbook use different palettes. ELA screens are built on a warm cream canvas with pale-blue zones; the K-5 math handbook uses a cool Gray/Blue token system. This guide follows the ELA templates for color, and borrows the handbook's rules and mechanics (layout per use case, lock-down procedure, "starting drop zone"). Don't paste math tokens into ELA screens.

§ Universal rules — true on every ELA screen

These apply regardless of interaction type. If a screen breaks one of these, fix it first.

  • The Polypad sits in the right column of the step; the prompt lives in the subtitle so the two components align vertically. ELA templates, Screen 2
  • Text and card sizes are not edited from the template. If your text overflows a card, reword it or choose a different template — never stretch the card. ELA templates
  • Titles above drop zones do not overlap the reset button. ELA templates
  • The action bar is locked down — see the profile below. No stray math/geometry tools, no eraser unless the task needs sketch. K-5 Handbook · Audit
  • Every draggable is Stay-in-Dropzones ON and not editable / not deletable.
  • The canvas uses one house background (never left transparent, never a random hand-typed hex).
  • Validation is either fully wired or intentionally absent — never a mix of validated and unvalidated answer zones on the same screen. Audit finding C3

The #b0bed2 vs #b0bed3 question — settled
Earlier the audit treated #b0bed2 as a one-digit typo of #b0bed3. The official template proves otherwise: both are intentional. #b0bed2 is the dashed answer-zone border; #b0bed3 is the solid source-zone fill and the card border. The real defect is when either value lands on the wrong role, or drifts to grey/near-black (#242436, #111111 as a zone fill).

The lock-down profile

Every ELA Polypad ships with the tool UI locked. The procedure (from the K-5 Handbook) is Disable All, then re-enable only what the task needs.

actionbar
"hidden" for drag-drop / free-response. Never the full math/geo enum.
toolbar
"hidden". Re-enable eraser/pen only for genuine sketch tasks.
grid
"none" for ELA content screens (templates ship gridless).
noPinchPan
true on fixed-viewport screens (K-3 find pan/zoom confusing).
noUndoRedo
true unless the task is step-by-step sketch.
delete on tiles
OFF — students must not be able to delete cards or the card bank.
K-5 Handbook The lock-down procedure: click Disable All, then toggle on only the tools the interaction needs. Source ↗
🛠 In a fix script
Apply the lock-down profile to every screen in an activity by copying a clean screen's options to all others: disable_all_via_api.py. The auto-fix pass also normalizes actionbar/toolbar to "hidden" and strips stray tool enums.

1  Card Match

Drag one card to the box beside each prompt. One answer per question, one-to-one.

When to use it
Use when there is exactly one correct answer per question — matching a quotation to a technique, a term to a definition, a cause to an effect.

The intended look

Intended style The official Matching template. Left: prompt cards on a pale-blue #d1e4f7 band. Center: dashed white answer zones (#b0bed2). Right: the source cards to drag. Clean cream canvas, no toolbar.  Open live ↗

Spec — true every time

Answer zone
#b0bed2 border, dashed, weight 2, white fill, layout:"center", cornerRadius:10, validation:"match"
Source/home zone
#b0bed3 solid fill, weight 2, layout:"center", validation:"none"
Cards
white fill, #b0bed3 border weight 1, cornerRadius:10, text #111111, editable:false, stayInDropzones:true
Zone count
One answer zone per prompt. Every answer zone validated — never 3 of 4.
Canvas
#fff8e9, grid none, actionbar + toolbar hidden
Validation
Wired after editing — each answer zone gets a captured storedSolution. A Check button is optional in ELA (many screens self-validate on drop).

Conformance checklist

  • Every answer zone has validation:"match" with a stored solution — none left none.
  • Answer zones are dashed #b0bed2; source zones are solid #b0bed3.
  • Cards are white with a #b0bed3 border, radius 10, not editable, Stay-in-Dropzones on.
  • No off-screen tiles; the initial viewport frames all content with margin.
  • actionbar & toolbar hidden; canvas #fff8e9.
  • Card text fits its card without resizing the card.

How to conform a non-standard screen

  1. Grab it. python grab_slide.py <activity-url> dumps every screen's polypadState.
  2. Classify zones. Any categorizer with validation match/compare, or the upper of a stacked pair, is an answer (drag-TO) zone; the rest are source (drag-FROM). The style_dropzones_via_api.py heuristic already does this.
  3. Restyle. Answer → dashed #b0bed2/white; source → solid #b0bed3. Set card fill/border/radius. Lock text (editable:false, stayInDropzones:true).
  4. Fill validation gaps. For any answer zone missing a solution, capture the correct card into storedSolution (see build_match_slide.py for the exports+wrapper-script pattern).
  5. Lock down + re-thumbnail. disable_all_via_api.py then regen_thumbs_fast.py.
🛠 In a fix script
One command does steps 2–5 across a whole activity: python audit_and_fix_style.py <url> --fix. It reports each deviation, applies the ELA palette, and re-thumbnails.

2  Card Sort

Drag many cards into a few category bins. Many-to-one; order inside a bin doesn't matter.

When to use it
Use when students organize information into 2 (sometimes 3) categories — sorting evidence, classifying examples, grouping by theme.

The intended look

Intended style The cleanest sort in the collection (from David Poras's demo): two dashed white category zones ("Telling" / "Showing") with a grey source bin of pale-blue cards below.  Open live ↗

Spec — true every time

Category zones
dashed border, white fill, #b0bed2, layout:"flow" (multiple cards land inside), radius 10
Source bin
one grey bin holding all cards, layout:"flow", padding 40 so cards space out; fill light grey or
Cards
consistent sizes, white or pale-blue fill, radius 10, cannot copy/delete, Stay-in-Dropzones on
Zone sizing
Zones sized for more answers than expected — sorting is many-to-one.
Canvas
cream/white, grid none, actionbar + toolbar hidden
Validation
wired after editing, or intentionally a self-check — applied uniformly.

Conformance checklist

  • Category zones use Flow layout (not Center) so multiple cards fit.
  • Exactly one source bin; all cards start inside it (or in home placeholders).
  • Cards are a consistent size and cannot be copied or deleted.
  • No leaked math/geo actionbar or eraser toolbar (a real defect even in the Categorize template).
  • If validated, every category zone is validated — no partial wiring.

Conforming vs. non-conforming (from the audit)

Conforming

Phineas Should've Been Dead — blue-dashed source → grey targets, match validation + check.

Open live ↗
Non-conforming

World of Chocolate — no validation, no check, full math toolbar exposed.

Open live ↗

How to conform a non-standard screen

  1. Confirm it's really a sort. If it's built as a free-form organizer but the content is a clean 2-category sort, rebuild it on the Categorize & Organize template (audit finding H1).
  2. Fix layout mode. Category zones must be layout:"flow", not center.
  3. Consolidate the source. One bin, Flow, padding 40; all cards inside with Stay-in-Dropzones on.
  4. Strip the toolbar. The Categorize template itself leaks geo-midpoint + eraser — set actionbar/toolbar to hidden.
  5. Lock down + re-thumbnail as above.
🛠 In a fix script
audit_and_fix_style.py classifies the bins, forces Flow on multi-card zones, wires Return-to-Dropzone on source cards, and strips the toolbar. See also refresh_dropzones.py for re-flowing after a resize.

3  Graphic Organizer

Drag labels or evidence onto the right region of a diagram — a Hero's-Journey wheel, a Venn diagram, a story map.

When to use it
Use when placement is spatial and meaningful — the diagram itself carries information and cards land in named regions. Often un-scored (spatial judgment) rather than match-validated.

The intended look

Intended style Hero's Journey Structure — an illustrated wheel on one tan background, labeled drop slots, and a clear Card Bank. The most cohesive organizer in the catalog.  Open live ↗

Spec — true every time

Diagram
a single background image, status:"locked" — students can't move it
Slots
categorizers positioned over regions; layout:"center" (one label per region) or "none" for free placement
Card Bank
one labeled source area; cards Stay-in-Dropzones on, delete removed from actionbar so the bank can't be deleted (audit finding H4)
Consistency
fixed vs. unlimited cards decided once and kept identical across copies (finding C5)
Canvas
one house background; no leaked toolbar; grid none

Conformance checklist

  • Diagram image is locked and can't be dragged or deleted.
  • The Card Bank and its cards can't be deleted (no delete in actionbar).
  • Every copy of the organizer uses the same fixed/unlimited card behavior.
  • One background color; no mixed zone styles or oversized headings.
  • Slots are visually consistent (same stroke/fill), not a mix of blue-dashed and grey-solid.

Conforming vs. non-conforming (from the audit)

Conforming

Hero's Journey — cohesive wheel + card bank on one tan background.

Open live ↗
Non-conforming

Venn Diagram — drawn as rectangles not circles, off-palette strokes, oversized heading, bare boxes.

Open live ↗

How to conform a non-standard screen

  1. Lock the furniture. Set the diagram image and Card Bank to status:"locked"; remove delete from the actionbar enum.
  2. Unify slot styling. Give every slot the same stroke/fill (dashed #b0bed2 for drop targets); purge near-black and off-palette blues/golds.
  3. Normalize the heading. One title size — don't leave a stray fontSize:5.
  4. Set one background and lock down the toolbar.

The auto-fix pass restyles categorizer slots to the ELA palette and locks text/furniture. Slot positioning over a diagram is layout-specific — do that by hand, then run regen_thumbs_fast.py.

4  Free Response

Type an answer into a box. No dragging — students write. The most consistent pattern in the catalog.

When to use it
Use when the answer is constructed text — definitions, short responses, notes. Two labeled columns with input fields, or a single passage + response.

The intended look

Intended style The Text-into-Polypad demo (David Poras): text pulled into Polypad via the CL sink, with input fields and passage components. This is the modern ELA free-response pattern.  Open live ↗

Spec — true every time

Input fields
question-blank tiles or CL input fields; non-editable prompt text, editable answer field
Text layouts
Auto-height (left-aligned, nothing fixed) · Fixed (center-aligned, width fixed) · Scrollable (left-aligned, W/H fixed) — pick one per box and keep it consistent
Text styling
author may set Fill, Stroke, Text color, Corner Radius, Drop Shadow
CL niceties
placeholder text via CL; resetOnChange when a field should clear on edit
Canvas
white, actionbar + toolbar hidden — the gold-standard lock-down

Conformance checklist

  • White background, everything hidden — borrow this options profile for other types too.
  • Prompt text is non-editable; only the answer field accepts input.
  • One text-layout mode per box (don't mix Fixed and Scrollable arbitrarily).
  • No categorizers, no stray script unless CL placeholder/reset is intended.

How to conform a non-standard screen

  1. Adopt the ELD/Text-into-Polypad options profile (white, all hidden) as the baseline — it's the cleanest in the whole catalog.
  2. Lock the prompts. Prompt text editable:false; only input fields editable.
  3. Pick one text layout per box and normalize color to a single value.
🛠 In a fix script
Free-response screens rarely need geometry fixes — the auto-fix pass mainly normalizes the options profile and text-tile lock status here.
Gold standard Responding to Passage 1 (ELD) — two labeled columns, four input fields, white background, everything hidden. Identical across hundreds of reuses. Open live ↗

5  Zoom Reader

Read or scroll through a sequence of panels; tap to zoom in. Passive — no answer to submit.

When to use it
Use for graphic-novel spreads, scavenger-hunt panels, or long illustrated passages where the student reads and zooms rather than manipulates.

The intended look

Intended style Odyssey graphic novel — nine image panels stacked in one tall column with a single zoom interaction. This pattern works, but has no shared template yet — each screen is hand-built.  Open live ↗

Spec — true every time

Panels
stacked image tiles in one column, locked
Interaction
a single zoom interaction; optional geo hotspots
Column width
define one column width and reuse it across every screen in the set
Canvas
one background; pan/zoom allowed here (this is the one type where pan is expected)

Conformance checklist

  • Panels are locked images in a single consistent-width column.
  • One zoom interaction pattern reused across the whole set (not re-invented per screen).
  • One background color across all screens in the sequence.
  • Pan/zoom intentionally enabled (unlike the fixed-viewport drag-drop screens).

How to conform a non-standard screen

This type has no shared template today — the fix is to define one and apply it across the set. Until then, the auto-fix pass will lock the images and normalize the background, but the column width and zoom wiring should be standardized by hand.

🛠 In a fix script
The auto-fix pass locks image tiles and sets one background; it does not touch the zoom interaction (type-specific). Flag Zoom Readers as needs a template.

6  Sentence Builder

Drag word tiles into a frame to build a sentence. Uses Polypad's letter tiles with built-in pronunciation.

When to use it
Use for sentence construction, word ordering, or fill-the-frame tasks — common in ELD English Building Blocks (EBB).

The intended look

Intended style EBB Compound Sentences — draggable letter word-tiles into Flow frames. The pattern is sound; it currently ships unvalidated.  Open live ↗

Spec — true every time

Word tiles
letter tiles (built-in pronunciation, voice) — strict schema: only name/text/voice/sense/POS/altPron
Frames
categorizer frames, layout:"flow" so words line up
Validation
decide self-check vs. graded; if graded, add match + a check button
Canvas
one background; toolbar hidden

Conformance checklist

  • Word tiles use the strict letter-tile schema (extra fields silently drop the tile).
  • Frames are Flow layout so multiple words fit in order.
  • Validation decision is explicit — either wired, or intentionally a self-check.
  • Toolbar hidden; one background.

How to conform a non-standard screen

  1. Decide graded vs. self-check. If graded, add match validation to the frame and a Check button (build_match_slide.py shows the words-mode wiring).
  2. Verify the letter-tile schema — strip any non-schema fields.
  3. Flow the frames and lock down the toolbar.
🛠 In a fix script
The auto-fix pass normalizes frame layout and toolbar; it will NOT invent validation (that's an authoring decision). It flags unvalidated builders for review.

§ Drop-zone deep dive

Drop zones (categorizers) are where most inconsistency lives. Get these right and half the catalog's problems disappear.

Use case → layout (from the K-5 Handbook)

The "starting drop zone" pattern

Cards should start inside an invisible home zone, not loose on the canvas. Turn Visible OFF, Stay-in-Dropzones ON, Center layout with a tile limit equal to the number of cards. This means a card is always in a zone — home or answer — so validation works either way, and click-to-place lights up every hover state (including the invisible home).

ELA template Grey home placeholders on the left hold the answer cards; the source bin at the bottom holds the draggables. Every card starts in a zone.

The key rule for ELA drag-drop: set an initial viewport that frames all content with ~1 tile of margin, and keep tiles on-screen — the audit found several screens relying on the viewport to pan back to content parked at large negative coordinates (finding F1).

No off-screen content
If customInitialViewport is doing the work of hiding tiles at x ≈ -892, the screen breaks the moment the viewport is slightly off. Bring content into positive coordinates.

§ The auto-fix script

A single pass that scans any activity for deviations from this guide and corrects them.

tools/audit_and_fix_style.py pulls an activity, checks every screen against the conformance checklists above, prints a per-screen report of what's off, and (with --fix) rewrites the polypadState to the ELA palette and lock-down, then re-thumbnails.

# scan only — report deviations, change nothing
python tools/audit_and_fix_style.py <activity-url>

# scan + fix + re-thumbnail
python tools/audit_and_fix_style.py <activity-url> --fix

The recurring offenders

  • Missing / hand-typed backgrounds — the audit found 22 distinct background values and 67 screens with none. Pick one house background.
  • Zone fill drifting to grey/near-black — #242436, #111111, #777777 as a zone fill is always wrong.
  • Partial validation — 3 of 4 answer zones wired. All or none.
  • Default "Button" label — the most common check-button label in the catalog was the un-renamed default. Use a real label if you keep the button.
  • Leaked full math/geo toolbar — 21 screens shipped the whole tool enum; students see irrelevant clutter. Lock it down.
  • Off-screen content propped up by customInitialViewport. Bring it on-screen.
